## 1. Two findings worth leading with

### 1.1 The ~73% plateau is overfitting, not underfitting

The first thing the new evaluation notebook produced answers TODO ┬¦7.1 step 2 directly.
Train-vs-val accuracy at each run's best epoch:

| run | arch | subset | train acc | val acc | gap |
|---|---|---|---|---|---|
| 1784397301 | bilstm | FP_118/xy | 0.9886 | 0.7526 | **0.236** |
| 1784395799 | bilstm | ME_126/xy | 0.9716 | 0.7503 | **0.221** |
| 1784396516 | bilstm | ME_132/xy | 0.9716 | 0.7476 | **0.224** |
| 1784393064 | gru | ME_126/xy | 0.9022 | 0.7472 | **0.155** |
| 1784399741 | lstm | ME_126/xy | 0.9225 | 0.7453 | **0.177** |

Every model memorizes the training set (90ŌĆō99%) and generalizes to ~75%. This is `train >> val`
by a wide margin, so the plateau task list's own branch condition says: **prioritize
augmentation and regularization (┬¦7.4) before adding features**. The normalization and motion
work (┬¦7.2/┬¦7.3) is still worth doing on its merits, but it is not what is capping accuracy
today ŌĆö a model that memorizes 99% of training data is not short of signal.

The BiLSTM gap being the largest is consistent: it has the most capacity (2.7M params) and the
weakest constraint (it can see the future), so it memorizes hardest.

| true ŌåÆ predicted | mean rate | reverse | | true ŌåÆ predicted | mean rate | reverse |
|---|---|---|---|---|---|---|
| awake ŌåÆ wake | 0.424 | 0.371 | | nap ŌåÆ sleep | 0.209 | 0.064 |
| mouth ŌåÆ lips | 0.309 | 0.226 | | that ŌåÆ stay | 0.205 | 0.149 |

| pencil ŌåÆ pen | 0.225 | 0.158 | | puppy ŌåÆ dog | 0.140 | 0.044 |

Combined with ┬¦1.1, the picture is: the models memorize training data almost perfectly, and
what they fail on at val time is dominated by semantically adjacent pairs. TODO ┬¦7.1 step 5

highest-value next diagnostic ŌĆö and it should explicitly allow the verdict "these two signs are
not reliably distinguishable in this data".

### meta.json schema v3 ŌĆö the `submission` block

Kaggle allows 100 submissions/day and every trained model should eventually be scored, so
"which models still need submitting" must be a query, not something remembered:

```sql
dataset = 'gislr' AND submission.tested = false   ORDER BY accuracy DESC   LIMIT 100
```

The block is deliberately **dataset-agnostic** ŌĆö `tested` means "scored on the
official/held-out test set", with `platform` distinguishing `"kaggle"` from `"local"`. No
Kaggle vocabulary in the required keys, so POPSIGN runs reuse it unchanged. `write_meta`
protects the block from the training loop's per-epoch rewrites (the same protection canonical
metrics already had), and `migrate_all` backfilled all **18 existing runs** to v3.

**Open**: `asl-signs` is a *code competition* ŌĆö the submit command takes `-k <kernel> -v
<version>`, i.e. submission goes **through a Kaggle kernel**, so each model's zip must be
attached to a kernel version. The submit cell therefore defaults to `DRY_RUN = True` and prints
the exact commands until that loop has been walked once by hand.

### The export route: ONNX abandoned, native Keras rebuild instead

The first submission attempt produced nothing, for a mundane reason (the queue skips runs with
no `submission.zip`, and export hadn't been run) ŌĆö but running export then exposed that the
**PyTorch ŌåÆ ONNX ŌåÆ `onnx2tf` ŌåÆ TFLite** route does not work for these models. Five distinct
failures, in order:

1. `onnx2tf` declares **no dependencies of its own** ŌĆö needs `tf-keras`, `onnx-graphsurgeon`,
   `sng4onnx`, `ai-edge-litert` at runtime.
2. `torch.nan_to_num`'s infinity handling emits an ONNX **`IsInf`** op it cannot convert.
3. It permutes 3D input layouts, emitting a model that wants `(T, 3, 543)`.
4. **Op coverage**: only **lstm** converted. **gru** died inside onnx2tf's own GRU handler
   (`tf.split(tR, 3)` on a 256-dim, expecting 768), **bilstm** on "mixed types to Tensor",
   **cnn1d** on a squeeze.
5. The one arch that converted then failed TFLite conversion on a malformed `Squeeze` onnx2tf
   itself generated.

Replaced with `modules/model/keras_export.py`: each architecture is rebuilt with **native Keras
layers** and the trained PyTorch weights transferred. All four now export, all under the cap:

| arch | tflite | keras parity | tflite parity |
|---|---|---|---|
| gru | 3.44 MB | 4.1e-06 | 3.8e-06 |
| lstm | 4.48 MB | 3.5e-06 | 3.8e-06 |
| bilstm | 11.06 MB | 3.2e-06 | 2.9e-06 |
| cnn1d | 2.89 MB | 6.1e-06 | 5.4e-06 |

This trades a *conversion* problem for a *weight-transfer* problem ŌĆö a gate-order or
bias-convention mistake would convert cleanly and predict garbage ŌĆö so export is gated on two
parity checks and writes nothing if either fails: the Keras rebuild vs `forward_full`, and the
**final .tflite file** vs PyTorch through the entire deployed path (raw `(T, 543, 3)` frames
with NaNs). Conventions that had to be handled: GRU gate reorder `[r,z,n]`ŌåÆ`[z,r,h]` with
`reset_after=True`, LSTM's two biases summed into Keras' one, LayerNorm epsilon forced to
PyTorch's 1e-5 (Keras defaults to 1e-3), Conv1d `(out,in,k)`ŌåÆ`(k,in,out)`.

The second gate immediately earned its keep, catching two bugs:

- **int8 quantization**: `tf.lite.Optimize.DEFAULT` (inherited from the original notebook code,
  "helps with the 40MB cap") shifted logits by ~1e-1. These models are 3ŌĆō11 MB against a 40 MB
  cap, so it bought nothing and cost accuracy ŌĆö now off by default.
- **Uninitialized resource variables**: saving the Keras-backed module directly produced a
  TFLite model that died at invoke with `READ_VARIABLE ... variable != nullptr` inside the RNN's
  WHILE loop. Weights are now frozen to constants; and since freezing via
  `from_concrete_functions` loses the output *name* the grader indexes by (`output["outputs"]`),
  the frozen function is re-wrapped in a module that re-declares the exact signature.

`onnx`, `onnx2tf`, `onnxruntime`, `onnx-graphsurgeon`, `sng4onnx`, `ai-edge-litert` and
`tf-keras` were all removed from `pyproject.toml`; `kaggle` added.

## 4. POPSIGN extraction confidence tuning (TODO ┬¦2.3)

New notebook **`src/popsign.0.dataset.confidence-tuning.ipynb`** + two modules
(`quality.py`, `overlay.py`). Sample: 50 videos = 5 classes ├Ś 10, seeded, globbed straight off
the raw video drives ŌĆö deliberately **not** dependent on the missing manifests (┬¦0.4/┬¦2.2), so
it is unblocked today. Sampling verified: 50 clips drawn from 124 available classes across all
four train parts.

### The finding that reshaped the notebook

The obvious grid ŌĆö sweep `min_hand_landmarks_confidence`, since the hands carry the sign ŌĆö is
**useless**. Driving each field to both extremes on one clip:

| field | 0.01 ŌåÆ 0.99 |
|---|---|
| `min_hand_landmarks_confidence` | **bit-identical output** (inert) |
| `min_pose_{detection,landmarks}_confidence` | hand detection rate **0.52 ŌåÆ 0.09** |
| `min_face_{detection,landmarks}_confidence` | face lost entirely at 0.99 (91% NaN) |

Holistic derives hand ROIs from the **pose** landmarks, so the pose thresholds are what
actually gate the hands. Sweeping the hand threshold would have produced a table of identical
rows and read as "tuning doesn't matter". The grid now sweeps pose thresholds (jointly and one
at a time) plus a face-off arm.

Also corrected against the real API: there is **no** `min_tracking_confidence` and no separate
hand *detection* threshold ŌĆö the actual field list is `extraction.CONFIDENCE_FIELDS`, and
unknown keys now assert rather than being silently ignored.

### Quality measurement

No ground truth exists for POPSIGN, so everything is a labelled **proxy**: per-group detection
rate, hand-presence rate, longest detection gap, frame-to-frame jitter, and rigid-bone
(shoulderŌåÆelbow) length CV. Composite `quality_score` z-scores them across the sweep with
tunable weights ŌĆö a *relative* ranking of the configs tried, not an absolute measure.

The visual test is weighted **80 worst / 10 median / 10 best** frames, because a bad config's
best frames still look fine; how it *fails* is the informative part. A detector that
confidently tracks the wrong region scores well on every proxy, which is exactly why the
numbers don't get the last word.

Chain smoke-validated end to end (2 configs ├Ś 2 videos ŌåÆ npz ŌåÆ metrics ŌåÆ overlays ŌåÆ contact
sheet). **Open caveat**: `pose_rate` was 1.0 in every arm including `min_pose_* = 0.99`, so
that proxy can't currently discriminate pose quality ŌĆö filed as a follow-up.

### First sweep results ŌĆö and why the sweep should pause

Two of the seven arms have now been extracted over the full 50-video sample (`default`
50/50, `pose_strict` 49/50). Written up in full at
**[docs/reports/confidence-tuning.md](../../reports/confidence-tuning.md)**; the short
version is that the thresholds are not the story.

**Thresholds barely move anything.** Paired over the 49 videos both configs extracted,
`pose_strict` (pose fields 0.7) costs 0.022 of any-hand detection rate versus `default`
(all fields 0.5) ŌĆö worse on 31 videos, better on 5 ŌĆö and buys 0.002 less hand jitter.
Consistent with ┬¦"the finding that reshaped the notebook": a stricter pose gate suppresses
marginal hand detections rather than improving them. `default` leads the composite score
(0.160 vs ŌłÆ0.163), but with two arms in the table that score is ┬▒1 by construction, so it's
directional only.

**The proxies are largely measuring clip padding.** Mean any-hand rate of 0.427 reads as
"MediaPipe fails on more than half the frames". It doesn't. Hand presence against normalized
clip position peaks at **0.86** mid-clip and sits at 0.12ŌĆō0.19 across the first and last
fifths; the median clip's first hand detection is at 27% of its duration and its last at 72%.
Measured between the first and last hand-bearing frame, the same extraction scores **0.85
mean / 0.94 median**. `longest_gap_frames` correlates with `n_frames` at **rho 0.84** ŌĆö the
"longest detection gap" proxy is very largely a measurement of the lead-in/lead-out.

So the effect being ranked (~0.02) is an order of magnitude below the artifact contaminating
the ranking (~0.4). Running the remaining five arms against these proxies would rank them
partly on clip-length variation within the sample. **Fix the measurement window first**
(restrict the proxies to the signing span), drop the constant `pose_rate` term, *then* run
the rest ŌĆö ~250 extractions, ~25 min at 19 workers.

**A separate deficiency, bigger than any threshold**: only **1.1%** of frames carry both
hands (left 9.5%, right 34.3%), and `hand_rate` tops out at exactly 0.50 across the whole
sample ŌĆö the signature of "exactly one hand, always". Several sampled signs (`car`, `bath`)
are two-handed in ASL. That is about *which* landmarks holistic returns, and no confidence
threshold in the grid addresses it. It needs the overlay frames looked at, not more tuning.

Downstream consequence worth carrying forward: any temporal model on POPSIGN will see ~50%
non-signing frames per clip. Trimming belongs in the POPSIGN feature-building stage.

## 4b. POPSIGN extraction driver ŌĆö it runs in the notebook after all (TODO ┬¦2.3)

This went through a full reversal in one day, and the reversal is the interesting part.

**First pass**: `popsign.0.dataset.extraction.ipynb` was given a CLI handoff
(`modules/scripts/extract_popsign.py`) on the strength of the pre-existing "multiprocessing
can't run in a Jupyter kernel on Windows" note, whose stated mechanism was *spawn re-imports
`__main__`, which in a notebook is the kernel*.

**That mechanism is wrong.** `multiprocessing.spawn.get_preparation_data` only sets
`main_path` when `__main__` has a `__file__` ŌĆö a kernel has none, so the child never
re-imports `__main__` at all. `sys_path` *is* propagated, and this project's worker functions
live in an importable module (`_extract_one`, `_init_worker`), not in a cell. The classic
Jupyter+spawn failure is about workers **defined in the notebook**; ours never were.
Confirmed by driving a real ipykernel over ZMQ: the pool ran 4 videos on 2 workers, 0 failed,
**21.6 s ŌĆö the same wall time as the CLI**, through the very resolution sequence that had
deadlocked. The guard is gone.

The hang that motivated it was real, but it is far better explained by the leaked-graph
deadlock above, which produces exactly that signature and which the pool cannot detect.

**The one objection that survived** was concrete and worth fixing rather than routing around:
MediaPipe logs from C++ (absl/glog) straight to file descriptor 2, and ipykernel captures
fd-level output into cell output by default (`IPKernelApp.capture_fd_output = True`). A
30K-video run would have written hundreds of `oneDNN`/`XNNPACK`/`inference_feedback_manager`
lines *per worker* into the `.ipynb` ŌĆö precisely the failure mode that once left a 17 MB
notebook in this repo (┬¦0.2). So `_init_worker` now redirects each worker's fd 2 to
`<out_dir>/<split>/_worker_stderr.log` before MediaPipe is imported: the messages stay
readable on disk and out of the notebook.

**Verified by executing the notebook's own cells** in a real ipykernel ŌĆö setup ŌåÆ manifest
generation ŌåÆ verification ŌåÆ pilot ŌåÆ ETA ŌåÆ cleanup:

| | |
|---|---|
| pilot trials | 2 (shrunk for the test), 0 failed |
| cell-output noise lines | **0** |
| total cell output | ~2 KB |
| worker logs | captured to `_worker_stderr.log` |

`extract_popsign.py` stays as an **optional** CLI for unattended runs that should outlive the
kernel. Both share the manifests and are resumable, so they are interchangeable.

The lesson worth keeping: a plausible mechanism attached to a real symptom is not a
diagnosis. The "can't run in Jupyter" story explained the observation, was repeated into four
documents and a runtime guard, and was wrong ŌĆö while the actual bug (a leaked native graph)
sat undetected until a pilot hung and the process table was actually looked at.

The chosen confidence config is now an explicit notebook constant threaded into the script
(`--confidence`), so the bulk run records which thresholds produced its output rather than
silently inheriting MediaPipe's defaults.

One bug worth recording because verifying the handoff cells *by printing them* could not
catch it: the shared options (`--confidence`, `--cpu-fraction`, `--ram-limit-pct`) were
defined only on the top-level parser, so the command the cells print ŌĆö which puts them after
the subcommand, as anyone naturally would ŌĆö died with `unrecognized arguments`. They are now
attached to the subparsers as well, with `SUPPRESS` defaults on the copies so a subparser
cannot clobber a value given before the subcommand. Both positions parse, and the
before/after override precedence is tested.

### The first pilot deadlocked ŌĆö leaked MediaPipe graphs

The first real pilot run stalled at **18/20 with every worker at 0% CPU** and no error. The
diagnosis, from live process inspection rather than from any traceback:

1. POPSIGN **mixes resolutions** ŌĆö 19 of the 20 trial videos are 1944├Ś2592, one (`drink`) is
   1080├Ś1920.
2. The worker keeps **one persistent `HolisticLandmarker`** across videos. The graph's
   segmentation-smoothing calculator compares each frame with the previous one, so the
   resolution change raises `INTERNAL: RET_CHECK failure ... current_mat->rows ==
   previous_mat->rows (1920 vs. 2592)` ŌĆö twice, entering and leaving that video.
3. The per-frame `except` handler rebuilt the landmarker by **rebinding `_LANDMARKER` without
   calling `.close()`**. Each `HolisticLandmarker` owns a native graph with ~70 threads, so
   every rebuild leaked one.
4. Evidence: the wedged worker sat at **219 threads / 1.3 GB** against 76 threads / 614 MB for
   its five siblings ŌĆö about three leaked graphs. It then stopped making progress entirely.
5. `imap_unordered` cannot detect this. The process is *alive*, just never returning, so the
   pool waits forever. Extrapolated to the real run, an overnight 30K-video extraction would
   have hung silently.

Fixes, all in `extraction.py`:

- **`_reset_landmarker()`** closes the old landmarker before building a new one ŌĆö the leak.
- **Proactive rebuild on resolution change**, checked once per video from
  `CAP_PROP_FRAME_WIDTH/HEIGHT`, so a mixed-resolution dataset no longer takes the per-frame
  exception path at all.
- **`maxtasksperchild=64`** recycles workers as a safety net, since a fresh process is the one
  guaranteed way to reclaim whatever a long-lived graph accumulated.

Verified on the exact failure sequence (2592 ŌåÆ 1920 ŌåÆ 2592, ending on the two videos that
hung): **4/4 extracted in 21.7 s, 0 failed**, frame counts and NaN fractions as expected.

Related, and fixed at the same time: `extract_popsign.py pilot` now **clears the temp tree
before benchmarking**. `extract_dataset` skips already-extracted videos, so leftover npz from
an interrupted pilot would have made a trial time ~0 videos and report a meaningless
throughput ŌĆö exactly the situation this stall left behind (18 npz on disk).

Two things this casts doubt on. First, the pre-existing "multiprocessing can't run in Jupyter"
diagnosis: `main_path` is only put in the child's preparation data when `__main__` has a
`__file__`, which a kernel lacks, so the child never re-imports `__main__`, and a `Pool` with
module-level workers *does* start (verified). The observed 30-minute hang was real but the
stated cause is wrong ŌĆö and a leaked graph wedges a worker in exactly the same silent way, so
that may have been this bug all along. The guard stays (running as a script is right
regardless: log volume, kernel restarts, multi-hour runs) but is now labelled empirical.
Second, `pose_strict` finishing at **49/50** in the confidence sweep is suspicious in the same
way, and is the likeliest explanation for that missing video.

## 6. Training consolidation ŌĆö four notebooks into one, hyperparameters into a config

The four `gislr.1.model.<arch>.ipynb` notebooks each carried their own `HYP` dict. Since "all
else identical" is the entire premise of the architecture comparison and the subset ablations,
keeping them in sync was both a chore and a correctness requirement ŌĆö a bad combination.

Replaced by **`src/gislr.1.models.training.ipynb`** (shared setup / config / split / feature
caches, then one section per architecture, then comparison + eval handoff) with every
hyperparameter in **`src/config/gislr.training.json`**, read at run time by
`modules/model/config.py`. Architectures inherit the `shared` block and may deviate only via an
explicit `overrides` entry, which ┬¦2 of the notebook prints ŌĆö so a deviation is visible instead
of being something you find by diffing four files. Validation rejects unknown architectures,
unknown per-arch keys, missing required hyperparameters, bad `coords`, a wrong schema version,
and ŌĆö the important one ŌĆö an override naming a key that isn't in `shared`, so a typo like
`lr_patiance` can't quietly become a setting that does nothing. All six failure modes tested.

### What the consolidation immediately caught

Diffing the working tree against `HEAD` before deleting the old notebooks turned up an
uncommitted change that explains ┬¦1.3:

**`cnn1d`'s `num_layers` had been changed from 5 to 2** as part of making all four `HYP` dicts
identical. For the recurrent models that parameter is the number of stacked layers, and 2 is
correct. For `CausalConv1D` it is the number of **dilated conv blocks**, which *is* the
receptive field:

```
receptive field = 1 + (kernel - 1) * ╬Ż 2^i
  5 blocks (dilations 1,2,4,8,16) ŌåÆ 125 frames   Ōēł MAX_SEQ_LEN = 128
  2 blocks (dilations 1,2)        ŌåÆ  13 frames
```

A 13-frame window on a 128-frame sequence cannot see a whole sign. The 0.54 scores are a
crippled model, not a verdict on 1D convolutions ŌĆö and the same well-meaning sync that unified
the hyperparameters is what broke it, which is precisely the failure mode the consolidation was

### The saved format

Recorded here because it is the contract every downstream POPSIGN stage depends on. One
`np.savez_compressed` archive per video, **the label carried by the path, not stored inside**:

```
<root>/data/raw/popsign/<split>/<label>/<video_id>.npz
```

| key | shape | dtype | meaning |
|---|---|---|---|
| `landmarks` | `(T, 543, 3)` | `float16` | normalized x, y, z ŌĆö **NaN where undetected** |
| `fps` | scalar | `float32` | source fps (30.0 assumed when cv2 reports 0) |
| `num_frames` | scalar | `int32` | `T` |
