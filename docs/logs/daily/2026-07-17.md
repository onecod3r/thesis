# Part I — Per-run metadata & the queryable index

## 1. The problem

Run folders (`src/models/<dataset>/<architecture>/<timestamp>/`) documented
each run well for a *human reader* (`README.md`, `data.md`) but not for a
*query*: finding the best N runs for a (dataset, architecture, subset)
combination meant opening READMEs one by one. The structured fragments that
did exist were inconsistent — the two oldest runs had `cache/eval_summary.json`
but no hyperparameter record, the 2026-07-16 runs had `cache/hyp.json` but no
eval numbers.

## 2. `metadata.json` — the per-run record

One `metadata.json` at each run folder root (schema_version 1): identity
(`run_id`, `dataset`, `architecture`, `model_name`, `streaming`), input
(`subset`, `n_landmarks`, `coords`, `feature_dim`), scale (`n_params`),
protocol (`split`, `n_val`, `n_classes`), provenance (`trained_date`,
`training_source`, `training_regime`, `epochs`, `best_epoch`,
`early_stopped`), results, and a nested `hyperparameters` dict.

Results carry an explicit **`eval_status`** so training-loop numbers can never
masquerade as leaderboard numbers:

| field | source | eval_status |
|---|---|---|
| `train_val_acc` | training loop (latest checkpoint history) | written at `"pending"` |
| `overall_accuracy`, `macro_accuracy`, `median_class_accuracy`, `n_classes_below_50pct` | `scripts/eval_gru.py` on the canonical split | promotes to `"canonical"` |

Backfilled 2026-07-17 for all existing gislr/gru runs (the two pre-`hyp.json`
runs reconstructed from their READMEs + `eval_summary.json`).

## 3. Lifecycle — who writes what

1. **Training notebooks** (§7 run-docs cell) write `metadata.json` alongside
   the `data.md`/`README.md` drafts — `eval_status: "pending"`. Re-running the
   cell preserves canonical-eval fields already filled in.
2. **`scripts/eval_gru.py`** (canonical per-class eval) updates the run's
   `metadata.json` in place and flips `eval_status` to `"canonical"`.
3. **`scripts/build_model_index.py`** (new; runs with CWD anywhere) flattens
   every `metadata.json` into **`src/models/index.csv`** (committed;
   hyperparameters as `hyp_*` columns) and answers filter queries directly:

```bash
.venv/Scripts/python.exe scripts/build_model_index.py                       # rebuild + full leaderboard
.venv/Scripts/python.exe scripts/build_model_index.py --dataset gislr --arch gru --top 3
.venv/Scripts/python.exe scripts/build_model_index.py --subset ME_126
```

The index's sort column `val_acc` = canonical `overall_accuracy` when present,
else `train_val_acc`; rows missing `metadata.json` are warned about, never
silently dropped. Rebuild after each training run or canonical eval.

# Part II — Fresh-run-folder policy

## 5. Before / after

`resolve_run_dir` in `gislr.1.model.gru.ipynb` used a per-subset pointer file
(`cache/gru_runs/<tag>.txt`) that made *any* re-run land in the existing
folder; `train_subset` then skipped completed runs. Retraining under the same
conditions was therefore impossible without manually deleting the pointer.

Now the pointer is followed **only while the pointed-to run is unfinished**
(auto-resume after an interrupt is unchanged, per the §1.2 resumability
convention — "finished" = early-stopped or epoch cap, from the checkpoint's
`finished` flag). A finished run is never reused: every new training start
creates a fresh `models/gislr/<arch>/<YYYYMMDD-HHMMSS>/` folder (timestamp =
training start) and repoints the pointer.

**Consequence to be aware of:** the training cell (§5 of the notebooks) is no
longer idempotent — re-running it retrains *every* subset in `RUN_SUBSETS`.
Trim `RUN_SUBSETS` when that isn't wanted (noted in the cell banner).

# Part III — Training regime v2 & the architecture-benchmark notebooks

## 6. Regime `v2-plateau-300` (TODO §3.2)

`gislr.1.model.gru.ipynb` now trains under a new regime (no run executed yet):

| knob | v1-onecycle-60 (all 8 existing runs) | v2-plateau-300 |
|---|---|---|
| batch size | 192 | **512** |
| lr | 3e-4 | **1e-3** (scaled with batch) |
| LR schedule | OneCycleLR (per-step) | **ReduceLROnPlateau** (val acc, factor 0.5, patience 5) |
| epochs | 60 fixed | **cap 300 + early stopping** (no val-acc gain > 1e-3 for 15 epochs) |

Design decisions:

- **OneCycleLR was dropped**, not truncated: it anneals over a fixed total
  epoch count, which early stopping would cut mid-anneal. ReduceLROnPlateau
  watches the same plateau signal as the stopper with shorter patience, so the
  LR gets `lr_patience` epochs to rescue a plateau before the run ends.
- **Resume-safety**: the plateau counter (`epochs_since_gain`), scheduler
  state and a `finished` flag persist in the every-epoch checkpoint — an
  interrupt never changes when a run would have stopped.
- **Comparability**: v1 and v2 runs are not hyperparameter-comparable;
  `training_regime` is recorded in checkpoints, `hyp.json`, `metadata.json`
  and `index.csv` (v1 backfilled). The canonical eval split/metric is
  unchanged and remains the only leaderboard requirement.

## 8. `scripts/eval_gru.py` generalization

The canonical eval script now serves every notebook: it dispatches the model
class on the checkpoint's `arch` key (`gru`/`lstm`/`bilstm`/`cnn1d`; absent →
gru for pre-2026-07-17 checkpoints) and feeds `xy` or `xyz` per the
checkpoint's `coords` key — which the 2026-07-17 xy runs needed (TODO §3.1's
"eval xy mode" prerequisite, now done). The model classes in the script must
stay byte-identical to the notebook definitions (noted in its docstring).
